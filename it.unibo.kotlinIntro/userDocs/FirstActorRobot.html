Introduction

The virtual robot introduced in it.unibo.virtualRobot2020\userDocs\LabVirtualRobot.html is just an example of a robot built according to some specific technology.
But, an important goal (non-functional requirement) of a software system designer should be that of making the code of a robot application as much independent as possible from the robot technology, in order to make it easier the substitution of a robot type with another.

Kotlin actors help us to achieve such a goal, by providing in a quite natural way a more abstract layer of reasoning.
Let us report, as an example, an actor written in Kotlin that 'provides services' for our it.unibo.virtualRobot2020\userDocs\LabVirtualRobot.html:

robotActor.kt

This file is also included in project it.unibo.kotlinintro, directory virtualRobotUsage.


In this solution, the robotActor.kt works as a service able to handle a set of commands represented by Strings. Its behavior is message-driven, since the actor executes actions related to the (staring part) of the first command received:
  • init: starts its job
  • end: terminates its job
  • move: sends a move-command to the robot, using the object virtualRobotSupport.kt as an utility
  • collision: 'reacts' to a collision message sent by virtualRobotSupport.kt when a robot-sonar detects an obstacle
  • sensor: 'reacts' to a sensor message sent by virtualRobotSupport.kt when a scene-sonar detects the robot

The Actor as a Finite State Machine

The behavior of the robotActor.kt can be modelled as a Finite State Machine:

	while( state == "working" ){
		var msg = channel.receive()
		val msgSplitted = msg.split('(')
		val msgFunctor  = msgSplitted[0]
		when( msgFunctor ){
			"init"      -> doInit()
			"end"       -> doEnd()
			"sensor"    -> doSensor(msg)
			"collision" -> doCollision(msg)
			"move"      -> doMove(msgSplitted)
			else -> println("NO HANDLE for $msg")
		}		
 	}
 	println("robotActor ENDS state=$state")


Each state is mapped into a function whose body specifies the actions to be executed in that state.

Application messages

While the robotActor.kt begins to look a 'service', its behavior is still strongly related to the JSON language of the virtual robot (concrete-robot interaction language or cril). To overcome such a further technology-dependency, let us introduce the definition of messages at application level: 

msg( MSGID, MSGTYPE, SENDER, RECEIVER, CONTENT, SEQNUM )

Moreover, let us delegate to the robot support the task of translating the application level in the cril and viceversa.

Towards more complex behavior

The robotActor.kt implements the business logic related to each message it can receive. However, since it is an active entity, we can now extend its behavior so to satisfy the first requirement of our Application0. For example,we can say that:
the robot starts to explore in a systematic and autonomous way the room after receiving the command 'explore' .
The problem to face now is the design and development of a behavior that msut be both proactive (the robot must perform the exploration activity in autonomous way) and reactive (while exploring, the robot should not lose its capability to handle other messages, for example end and collision).
